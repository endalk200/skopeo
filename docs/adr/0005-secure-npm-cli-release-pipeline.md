# Secure npm CLI Release Pipeline

Skopeo publishes one public npm package, `@skopeo/cli`, which provides the `skopeo` executable; internal workspace packages remain implementation details and are bundled into the CLI artifact rather than published separately. Releases use a Changesets-driven release PR, a separate locked-down staging workflow, GitHub Actions OIDC trusted publishing, and npm staged publishing with stage-only trust permissions so CI can stage but cannot directly publish a public package.

The release artifact is a Node-targeted bundled JavaScript CLI built by Bun from `apps/cli/src/bin.ts`, with Release Metadata stamped into the bundle at build time. The publish workflow must run the full quality gate, build the bundle, inspect the npm tarball allowlist, install the packed tarball into a fresh project, smoke-test the installed `skopeo` command, and only then run `npm stage publish`; final public availability requires npm-side maintainer approval with 2FA.

The first registry entry is a one-time manual 2FA bootstrap publish because npm staged publishing requires the package to already exist. After bootstrap, `@skopeo/cli` must require 2FA and disallow token publishing, trust only the GitHub Actions staging workflow and `npm-production` environment, allow `npm stage publish` but not direct `npm publish`, and use GitHub Environment approval before staging. Successful staging creates a `skopeo-cli-vX.Y.Z` tag and draft GitHub Release; maintainers publish the GitHub Release only after approving the staged npm package.

Release governance treats `main` protection, CODEOWNERS review for release-sensitive files, pinned third-party GitHub Actions, least-privilege workflow permissions, and no privileged `pull_request_target` workflows as prerequisites for npm distribution. Runtime-relevant moderate-or-higher vulnerabilities block release unless explicitly risk-accepted, while dev-only low/moderate advisories are tracked but do not automatically block publication.
