# Config-Driven Model Providers

Status: Accepted

Amends: ADR 0003 (extends the Skopeo Configuration schema), ADR 0007 (relaxes profile selection and adapter construction)

A Model Provider is the access channel through which the Code Review Agent reaches a model — the model and the channel are independent axes: `gpt-5.5` is one model, while OpenAI, OpenRouter, and a corporate AI gateway are three ways to reach it. Skopeo Configuration gains a named provider registry (`[providers.*]`), a per-model routing table (`[models.*]`), and Default Review Profile selection (`[review]`), all defined in `@skopeo/config` next to the existing telemetry and devtools settings. A new `@skopeo/providers` package owns a `ModelProviderService` that resolves a model to a ready TanStack AI adapter by consulting the parsed configuration; `@skopeo/config` stays free of TanStack dependencies, mirroring the boundary ADR 0006 drew for Agent Tools.

The registry is keyed by name. `openai`, `anthropic`, and `openrouter` are reserved well-known entries with a fixed dedicated adapter each (`@tanstack/ai-openai`, `@tanstack/ai-anthropic`, `@tanstack/ai-openrouter`) and an overridable `base_url` (Azure OpenAI, mimicking proxies). Any other name declares a custom provider: `base_url` is required and a `protocol` field (`openai`, the default, or `anthropic`) selects the adapter. Within the `openai` protocol two wire dialects exist, and the split is deliberate: the well-known `openai` provider speaks the official Responses API (the richest surface for GPT reasoning options), while custom `openai`-protocol providers speak Chat Completions — the de-facto meaning of "OpenAI-compatible" among gateways, verified live: a LiteLLM→Azure gateway rejects the Responses API's streaming options, and ollama/vLLM do not serve `/v1/responses` at all. A `base_url` override changes the endpoint, never the dialect: the well-known `openai` entry keeps the Responses API even when overridden, so the override target must serve `/v1/responses` (official mirrors, Azure's `v1` API surface); a Chat-Completions-only endpoint must be declared as a custom provider instead. Silently downgrading the dialect on override was rejected because it would strip Responses-only reasoning options from mirrors that do support them — the same silent-knob-dropping this ADR forbids for protocol mismatches. The resolved wire dialect is passed to Review Profiles alongside the adapter, because vendor option shapes differ per dialect (Responses nests `reasoning.effort` and `text.verbosity`; Chat Completions takes flat `reasoning_effort` and `verbosity` — and the Responses API rejects top-level `verbosity` outright). Routing ships with code defaults (each model to its vendor) which config overrides per model, with an optional provider-specific `model_id` for gateways that rename models. A single custom slot was rejected because real setups combine a corporate gateway with local endpoints, and per-model routing needs distinguishable names; implicit provider-priority routing was rejected because behavior would depend on config ordering.

Each code-defined profile model declares its required wire protocol, and routing a model to a provider that cannot speak it is a hard validation error — never silent knob-dropping or best-effort translation. A Review Profile is a tuned bundle (ADR 0007); stripping `claude-opus-4-8`'s Anthropic-specific reasoning settings to squeeze it through an OpenAI-shaped endpoint would deliver a different, shallower review than the user selected while still charging for it. OpenRouter's adapter translates natively and is therefore protocol-compatible with all models. Rejection is also the reversible direction: relaxing to warn-and-drop later is backward-compatible, while tightening would break configs.

Provider credentials come from environment variables only: well-known defaults (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `OPENROUTER_API_KEY`) plus an `api_key_env` field naming an alternative variable. A plaintext `api_key` field in the TOML file is rejected with a hard validation error — the deliberate stance is that `~/.skopeo/config.toml` is always secrets-free, so it can be committed, shared, and printed by diagnostics without redaction concerns. All resolved keys flow through Effect `Config.redacted`, are optional for custom providers (local endpoints need none), and are required lazily — only when a routed request actually reaches the provider, preserving the pre-existing deferred-key behavior. Keychain integration and OAuth flows were rejected as heavy machinery with no current need; `api_key_cmd` for secret managers is a tracked fast-follow (issue #36).

Validation extends the existing `ConfigValidationReport`: structural facts about the file are hard errors (unknown provider or model names, missing `base_url`, invalid or forbidden `protocol`, protocol-compatibility violations, plaintext keys), while environment- and intent-dependent facts are warnings (no resolvable key for the effective profile's provider, declared providers nothing routes to). Unknown model names are errors rather than forward-compatible warnings because silently ignoring a routing rule sends requests to the wrong provider, which is worse than a version-mismatch failure. Environment overrides exist for `[review]` only (`SKOPEO_REVIEW_MODEL`, `SKOPEO_REVIEW_DEPTH`); flattening the provider registry and routing table into environment-variable grammar was rejected as an encoding swamp with no use case, since provider topology is per-machine-stable and credentials are already env-native.
